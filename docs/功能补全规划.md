# 桌面客户端有但我们缺的功能

按实现难度排序，标注每个功能的可行性和所需工作量。

---

## 简单

### 联系人详情页

**可行性**: 高  
**工作量**: 1-2 小时

点击联系人/会话中的头像或昵称，弹出详情面板：
- 头像大图、昵称、备注、微信号
- 首次聊天时间、最近聊天时间
- 消息统计（总条数、类型分布）
- 共同群聊列表

数据来源：contact 数据库 + 消息统计查询，后端已有大部分接口。

### 表情包浏览与管理

**可行性**: 高  
**工作量**: 1-2 小时

独立页面，展示所有使用过的表情包：
- 按使用频率排序
- 按会话来源分组
- CDN 表情直接展示，本地表情尝试解密显示
- 网格布局，支持放大预览

数据来源：消息表 type=47 的 emojiUrl 字段，已有数据。

### 群聊成员头像与昵称完善

**可行性**: 高  
**工作量**: 已部分完成

当前状态：群聊消息已显示头像，但需配置 SELF_WXID。
- 设置页面优化：从通讯录列表选择自己而非手动输入 wxid
- 成员面板点击可跳转到私聊

## 中等

### 合并转发消息完整展示

**可行性**: 高  
**工作量**: 2-3 小时

当前状态：后端已提取全部子消息到 subMessages 字段，前端有基础折叠面板。
- 弹窗模式展示完整合并转发内容
- 支持逐条滚动浏览
- 保留原始发送者信息
- 长合并转发支持搜索

### 语音消息播放

**可行性**: 中（依赖 silk 解码）  
**工作量**: 3-5 小时

微信语音消息使用 Silk 编码：
- 方案 A：Python silk-decoder 转换为 WAV/MP3，前端 HTML5 audio 播放
- 方案 B：WebAssembly silk 解码器浏览器端直接播放
- 需要定位语音文件的实际存储路径和解密方式（如有加密）
- 语音时长显示已有，转文字显示已有

### 消息引用/回复链展示

**可行性**: 高  
**工作量**: 2-3 小时

当前状态：回复消息显示被引用内容，但不可交互。
- 点击引用内容跳转到原始消息
- 显示完整回复链（A 回复 B 回复 C）
- 引用内容中的图片/链接可预览

### 图片/视频浏览增强

**可行性**: 高  
**工作量**: 2-3 小时

- 图片灯箱支持左右滑动切换
- 按日期分组浏览某会话的所有图片（类似朋友圈九宫格）
- 视频缩略图（首帧）
- 原图下载

### 消息状态标记

**可行性**: 中  
**工作量**: 2-3 小时

模拟桌面客户端的消息状态：
- 已读/未读标记（本地记录）
- 置顶会话
- 消息免打扰标记

## 困难

### 位置消息地图展示

**可行性**: 中（需要地图 SDK）  
**工作量**: 4-6 小时

- 接入高德/腾讯地图 JavaScript API
- 解析位置消息中的经纬度
- 地图上标记并展示位置信息
- 地图 API Key 需要用户自行申请或内置

### 实时数据同步

**可行性**: 低  
**工作量**: 5-8 小时

当前靠定时刷新（60s），桌面客户端是实时的：
- 方案 A：监听数据库文件变化（fs.watch）触发增量解密
- 方案 B：WebSocket 推送变更到前端
- 微信不实时写盘，未打开的会话最新消息可能缺失，这是底层限制
- 建议保持当前方案，优化为"打开会话时自动刷新该会话数据"

### 小程序/视频号内容展示

**可行性**: 低  
**工作量**: 大

- 小程序消息：解析 XML 提取标题/描述/封面图
- 视频号消息：可能需要额外的 API 调用获取内容
- 桌面客户端也未必能完整展示，优先级低
